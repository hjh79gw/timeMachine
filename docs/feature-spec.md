# 타임머신 모의투자 - 기능 명세서

---

## 1. 서비스 개요

### 1-1. 서비스 목적
- 과거 실제 주식 시장 데이터를 기반으로, 그룹 멤버끼리 투자 실력을 겨루는 턴제 멀티플레이 시뮬레이션 게임

### 1-2. 기술 스택
| 구분 | 기술 |
|------|------|
| 백엔드 | Java, Spring Boot |
| 프론트엔드 | React, TypeScript |
| 데이터베이스 | PostgreSQL |
| API 형식 | REST API (JSON) + WebSocket (STOMP) |
| 외부 API | 한국투자증권(KIS) Open API, OpenAI GPT API |
| 크롤링 | Jsoup (네이버 금융 뉴스) |

### 1-3. 통신 방식 구분
| 구분 | 방식 | 대상 |
|------|------|------|
| 주문 (매수/매도) | REST | 개인 → 서버 → DB 저장 |
| 차트 조회 | REST | 개인 요청 |
| 종목 검색 | REST | 개인 요청 |
| 게임 이력 조회 | REST | 개인 요청 |
| 턴 이동 투표 | WebSocket | 전원 브로드캐스트 |
| 턴 전환 데이터 (AI 브리핑, 랭킹) | WebSocket | 전원 브로드캐스트 |
| 개인별 잔고/보유종목 갱신 | WebSocket | 개인별 전송 |
| 대기실 참여자 변동 | WebSocket | 전원 브로드캐스트 |
| 게임 시작 알림 | WebSocket | 전원 브로드캐스트 |

- **REST**: 개인 행동 (주문, 조회) → 즉시 DB 저장/조회
- **WebSocket**: 공유 이벤트 전파 (턴 전환, 투표, 랭킹 등) → 실시간 브로드캐스트

### 1-4. 핵심 특징
- **과거 시장 재현**: 한국투자증권(KIS) API로 수집한 실제 과거 일봉 데이터 기반
- **AI 시장 브리핑**: 네이버 금융 뉴스 크롤링 + GPT 분석으로 해당 날짜의 실제 시장 동향 제공
- **그룹 경쟁**: 같은 그룹 멤버끼리 동일한 시장 조건에서 투자 → 수익률 랭킹
- **AI 최종 분석**: 게임 종료 시 시장 동향 대비 매매 적절성 평가

### 1-3. 전체 구조
- 유저는 외부 서비스에서 그룹에 소속
- 같은 그룹 멤버끼리만 플레이 가능
- 그룹당 활성화된 게임 방은 최대 1개

## 2. 주식 데이터 수집 (KIS API 배치)

### 2-1. 개요
- 한국투자증권 Open API를 통해 국내 주식 과거 일봉(OHLCV) 데이터를 DB에 수집
- 수집된 데이터는 게임의 기반 데이터로 사용 (현재가, 차트, 체결가 등)
- 관리자 전용 API로 수동 실행

### 2-2. KIS API 사양
| 항목 | 내용 |
|------|------|
| API | 국내주식 기간별 시세 (FHKST03010100) |
| 한 번 호출당 최대 건수 | 100건 |
| 호출 제한 | 초당 10건 이내 (계좌당) |
| 응답 데이터 | 시가, 고가, 저가, 종가, 거래량, 등락률 |

### 2-3. 서버 부하 방지 전략
| 전략 | 구현 내용 |
|------|-----------|
| 요청 간 딜레이 | 각 API 호출 사이 300ms sleep |
| 종목 간 딜레이 | 종목 완료 후 1초 대기 |
| 분기별 분할 호출 | 100건 제한 대응 → 3개월 단위로 분할 요청 |
| 429 대응 | exponential backoff (2초 → 4초 → 8초), 최대 3회 재시도 |
| 동시 실행 방지 | AtomicBoolean으로 배치 중복 실행 차단 |
| 비동기 실행 | @Async로 별도 스레드에서 실행, 서버 블로킹 방지 |

### 2-4. KIS API 운영 주의사항 (한투 공지 기반)
- API 호출 시 무한 등록/해제 반복 금지 (구독 Set 변경은 명시적 요청 시에만)
- 과도한 호출 시 IP 차단 가능성 있음 → 반드시 딜레이 준수
- 토큰 만료 시 자동 재발급 필요 (매일 01:00 스케줄러로 갱신)
- WebSocket 실시간 시세는 구독 종목이 없으면 호출하지 않음

### 2-5. 배치 관리 API
| API | 설명 |
|-----|------|
| POST /api/game/admin/batch/init | 배치 초기화 (종목 목록 + 기간 설정) |
| POST /api/game/admin/batch/run | 배치 비동기 실행 |
| GET /api/game/admin/batch/status | 배치 진행 상태 조회 (PENDING/IN_PROGRESS/DONE/FAILED) |

### 2-6. 데이터 저장 구조
- **game_stock_info**: 종목 코드, 종목명, 시장(KR), 섹터
- **game_stock_daily**: 종목 코드, 거래일, OHLCV(시가/고가/저가/종가/거래량), 등락률
- **game_batch_progress**: 종목별 배치 상태, 마지막 수집일, 재시도 횟수, 에러 메시지
- UPSERT 방식으로 중복 데이터 자동 갱신 (ON DUPLICATE KEY UPDATE)

## 3. 로비 (게임 진입점)

### 3-1. 활성 게임 없는 상태
- "방 만들기" 버튼 표시
- 중앙에 "활성화된 게임이 없습니다" 안내 + 아이콘
- 하단에 "과거 게임 이력" 카드
  - 각 항목: 시드머니, 기간(3M/6M/1Y), 게임 날짜 범위, 최종 수익률, 상태(완료)
  - "전체보기" 링크

### 3-2. 활성 게임 있는 상태
- "방 만들기" 버튼 숨김
- "현재 활성화된 게임" 카드 표시
  - 상태 뱃지 (대기중 / 진행중)
  - 방장 이름
  - 현재 참여 인원 수
  - "입장하기" 버튼
- 하단에 "과거 게임 이력" 동일하게 표시

## 4. 방 만들기 (게임 설정)

### 4-1. 설정 항목
| 항목 | 옵션 | 기본값 | 설명 |
|------|------|--------|------|
| 시드머니 | 500만 / 1,000만 / 3,000만 | 1,000만 | 모든 참여자에게 동일 지급 |
| 게임 기간 | 3개월(3M) / 6개월(6M) / 1년(1Y) | 6M | DB에 데이터 있는 범위 내에서 랜덤 시작일 결정 |
| 이동 일수 | 5일 / 7일 / 10일 | 7일 | 한 턴에 몇 영업일 이동할지 |
| 난이도 | 초보 / 고급 | 초보 | (추후 난이도별 기능 차별화 예정) |

### 4-2. 시작일 결정 로직
- DB에 저장된 일봉 데이터의 거래일 범위에서 랜덤 선택
- 시작일 + 게임 기간이 데이터 범위를 초과하지 않도록 필터링
- 모든 참여자가 동일한 시작일/종료일로 플레이

### 4-3. 제약 조건
- 그룹당 동시에 1개의 방만 생성 가능
- 방 생성자 = 방장(HOST)
- 방 생성 시 게임 상태: **WAITING**

## 5. 대기실

### 5-1. 헤더 영역
- 뒤로가기 버튼 → 로비로 복귀
- "대기실" 타이틀 + "멤버들이 모두 모이면 게임을 시작하세요" 안내
- 우측 상단: 게임 설정 요약 (시드 / 기간 / 난이도)
- 상태 뱃지 (대기 중)

### 5-2. 참여자 패널 (좌측)
- "참여자 (현재/최대)" 표시 (예: 참여자 3/6)
- 참여자 목록
  - 번호, 이름, 역할 (HOST 뱃지는 방장만), 준비 상태 (준비 완료)
- 그룹 멤버가 로비에서 "입장하기" 클릭 시 실시간으로 목록에 추가
- 참여자 퇴장 시 목록에서 실시간 제거
- 퇴장 후 재입장 가능

### 5-3. 그룹 채팅 패널 (우측)
- 다른 팀원이 개발하는 그룹 채팅 기능을 임베드 예정
- 채팅 완성 전까지는 빈 영역에 "채팅 연동 예정" 플레이스홀더 표시
- 별도 임시 채팅 구현하지 않음

### 5-4. 게임 시작
- 하단 전체 너비 "게임 시작하기" 버튼
- **방장만** 클릭 가능
- **최소 2명** 이상 참여 시에만 활성화
- 시작 시 모든 참여자 동시에 게임 플레이 화면으로 전환
- 게임 상태: **WAITING → IN_PROGRESS**
- 각 참여자에게 시드머니 지급 + 첫 번째 턴 생성

## 6. 게임 플레이

### 6-1. 턴 시스템
- 게임은 턴제로 진행
- 각 턴 = 설정한 이동 일수만큼 영업일 기준으로 시간 이동
- **모든 참여자가 같은 턴(같은 날짜)을 공유** — 개인별 독립 진행 아님
- 다음 날짜가 게임 종료일을 초과하면 자동 종료

### 6-2. 턴 이동 투표
#### 투표 시작
- **방장만** "다음 턴" 버튼 클릭 가능
- 클릭 시 **방장의 찬성 1표 자동 반영** + 전원에게 투표 알림 (웹소켓)

#### 투표 진행
- 참여자들에게 찬성/거절 투표 UI 표시
- 제한시간: **30초**

#### 투표 UI 구현 전략
- 투표 컴포넌트(`TurnVotePanel`)를 **독립 컴포넌트**로 구현
- **1단계 (채팅 미완성)**: 웹소켓 투표 이벤트 수신 시 화면 중앙 모달로 표시
- **2단계 (채팅 연동 후)**: 채팅 메시지 타입에 `VOTE` 추가 → 채팅 말풍선 내부에 투표 컴포넌트 임베드
- 투표 로직(웹소켓 통신, 상태 관리, 타이머)은 동일하며, 표시 위치만 변경

#### 투표 즉시 종료 조건 (둘 중 하나 충족 시)
1. **찬성 과반수 도달** → 즉시 턴 이동
2. **전원 투표 완료** (찬성+거절 합산 = 전체 인원) → 과반수 판단

#### 과반수 기준: ceil(참여 인원 / 2)
| 인원 | 필요 찬성 수 |
|------|-------------|
| 6명 | 3명 |
| 5명 | 3명 |
| 4명 | 2명 |
| 3명 | 2명 |
| 2명 | 1명 |

#### 투표 결과 처리
| 상황 | 결과 |
|------|------|
| 과반수 찬성 (즉시 종료 또는 타이머 만료 시) | 턴 이동 실행 |
| 과반수 미달 (전원 투표 완료 또는 타이머 만료) | 투표 취소, 현재 턴 유지 |

#### 턴 이동 실행 시 처리 (서버에서 1회 실행)
1. 모든 참여자의 보유 종목 현재 턴 날짜 기준으로 시가 평가
2. 모든 참여자의 포트폴리오 스냅샷 저장 (자산 변동 그래프용)
3. 모든 참여자의 자산/수익률 업데이트 + 랭킹 계산
4. 현재 턴 COMPLETED 처리
5. 다음 영업일로 새 턴 생성
6. AI 브리핑 조회/생성 (서버에서 1회만 실행)
   - **① DB 조회**: 해당 날짜의 AI 브리핑이 DB에 존재하는지 확인
   - **② 캐시 히트**: 이미 존재하면 DB에서 브리핑 데이터 그대로 반환 (크롤링/AI 호출 없음)
   - **③ 캐시 미스**: 존재하지 않으면 →
     - a. 해당 날짜 뉴스 기사 DB 조회 → 없으면 네이버 금융 크롤링 → 크롤링 결과 DB 저장
     - b. 기사 데이터를 GPT API에 전달 → AI 브리핑 생성
     - c. 생성된 브리핑 DB 저장 (다음 조회 시 캐시로 활용)
7. 웹소켓으로 전원에게 일괄 전송:
   - 공통: 새 턴 정보(날짜, 턴번호), AI 브리핑, 랭킹
   - 개인별: 각 유저의 잔고, 보유종목 평가

### 6-2. 주문 (매수/매도)

#### 통신 방식
- **REST API**로 처리 (`POST /api/game/turns/{turnId}/orders`)
- 주문 즉시 **DB에 저장** (서버 메모리 캐시 사용 안 함)
- 사유: 서버 재시작 시 데이터 유실 방지, 게임 종료 분석용 데이터 보장, 서버 메모리 부담 방지
- 재접속 시 DB에서 매매내역 조회하여 복원

#### 종목 선택
- 종목 검색: 종목명 또는 종목 코드 입력 → 실시간 검색 결과 드롭다운
- 선택 시 해당 종목의 당일 시세(종가) 표시

#### 주문 유형
| 유형 | 체결 조건 | 체결 가격 |
|------|-----------|-----------|
| 시장가 매수 | 즉시 체결 | 해당일 시가 |
| 시장가 매도 | 즉시 체결 | 해당일 시가 |

- **시장가 주문만 지원** (지정가 미지원)
- 분봉 차트는 시세 흐름 참고용으로만 표시되며, 실제 매수/매도는 시장가(시가)로만 체결됨
- 주문 패널에 "모든 주문은 해당일 시가로 체결됩니다" 안내 문구 표시

#### 비용
| 항목 | 비율 |
|------|------|
| 매수 수수료 | 0.015% |
| 매도 수수료 | 0.015% |
| 매도 세금 | 0.18% |

#### 제약 조건
- 잔고 부족 시 매수 거부 (필요 금액 = 체결가 x 수량 + 수수료)
- 보유 수량 부족 시 매도 거부
#### 수량 선택 보조
- 10% / 25% / 50% / 100% 퍼센트 버튼 (보유 현금 기준 최대 매수 수량 자동 계산)
- +/- 버튼으로 수량 조절

### 6-3. 나비 시장 동향 브리핑 (AI)

#### 뉴스 데이터 수집 전략
- **사전 배치 수집**: 출시 전 최근 5년치 뉴스를 미리 크롤링하여 DB에 저장
- **게임 가능 기간**: 최근 5년 이내로 제한 (사전 수집된 데이터 범위)
- **게임 중에는 DB 조회만** 발생 (크롤링/GPT 호출 거의 없음)
- 혹시 누락된 날짜가 있을 경우에만 on-demand 크롤링 실행

#### 섹터별 뉴스 크롤링
- **섹터 10개 × 키워드 검색 × 상위 3건 = 총 30건**
- 네이버 금융 뉴스 키워드 검색 API 활용:
  `https://finance.naver.com/news/news_search.naver?q={키워드}&pd=4&stDateStart={날짜}&stDateEnd={날짜}`

| 섹터 | 검색 키워드 |
|------|-----------|
| 반도체 | 반도체 |
| IT/소프트웨어 | IT 소프트웨어 |

- 기사 본문: `https://n.news.naver.com/mnews/article/{officeId}/{articleId}` → `#dic_area` (전문 수집)
- 크롤링 부하 방지: 요청 간 300~500ms 딜레이, User-Agent 헤더 설정

#### 뉴스 데이터 조회 흐름 (게임 중 턴 전환 시)
1. DB에서 해당 날짜 뉴스 기사 조회
2. **있으면**: DB 데이터 그대로 사용 (크롤링 안 함) — 정상 케이스
3. **없으면**: 네이버 금융 크롤링 실행 → 결과 DB 저장 — 예외 케이스 (누락 날짜)

#### AI 브리핑 조회/생성
- **처리 흐름**:
  1. DB에서 해당 날짜 AI 브리핑 조회
  2. **있으면**: DB 데이터 그대로 반환 (GPT 호출 안 함)
  3. **없으면**: 기사 30건 전문을 GPT에 전달 → 시장 동향 브리핑 생성 → DB 저장
- **AI 모델**: gpt-4o-mini (턴마다 호출, 기사 요약/분류 수준이라 mini로 충분)
- 턴 테이블(game_turn)의 ai_briefing 컬럼에도 저장 (게임 종료 분석용)

#### AI 브리핑 프롬프트 설계
- **시스템 프롬프트**:
  - AI 이름: "나비" (주식 투자 시뮬레이션 게임의 시장 동향 해설자)
  - 제공된 뉴스 기사 내용만 근거로 분석
  - **GPT 사전 학습 지식 사용 금지** (미래 정보 유출 방지)
  - 이모티콘 사용 금지
- **유저 프롬프트**:
  - 기사 30건을 섹터 태그와 함께 전달 (예: `[반도체] 제목: ... 내용: ... URL: ...`)
  - 기사에 없는 내용 추측/추가 금지
  - 주요 이슈에 해당 기사 원본 URL 첨부
- **max_tokens**: 1500

#### 브리핑 응답 형식
```
[시장 개요] (4~5문장, 전체 시장 동향을 구체적 수치와 함께 상세히 요약)
[주요 이슈]
- 가장 영향력 있는 이슈 + 근거 기사 URL
- 두 번째 영향력 있는 이슈 + 근거 기사 URL
- 세 번째 영향력 있는 이슈 + 근거 기사 URL
[투자 힌트] (1~2문장, 특정 종목명 언급 금지)
```
- 주요 이슈는 AI가 30건 기사를 종합 분석하여 **시장 영향력이 큰 3개만 선별**
- 모든 섹터를 나열하지 않음 — 시장 개요와 투자 힌트에서 전체 동향 커버

#### 사전 배치 크롤링 (출시 전 1회 실행)
- 최근 5년간 영업일 기준 뉴스를 섹터별로 크롤링
- 크롤링 완료 후 각 날짜별 AI 브리핑도 사전 생성하여 DB 저장
- 이후 게임 중에는 DB 조회만으로 즉시 응답 가능

### 6-4. 차트
- 종목 선택 시 해당 종목의 캔들차트 표시 (lightweight-charts 라이브러리)
- 현재 게임 날짜 이전 데이터만 표시 (미래 데이터 노출 금지)

| 차트 모드 | 데이터 범위 | 데이터 소스 |
|-----------|-------------|-------------|
| 일봉 | 시작일 ~ 현재 턴 날짜 | DB 일봉 데이터 직접 사용 |
| 주봉 | 현재 날짜 기준 2년 전 ~ 현재 | DB 일봉 → 서버에서 ISO 주 기준 집계 |
| 월봉 | 현재 날짜 기준 5년 전 ~ 현재 | DB 일봉 → 서버에서 월 기준 집계 |
| 분봉 (시뮬레이션) | 해당일 09:00~15:30 | 일봉 OHLCV 기반 시뮬레이션 생성 (seed 기반 일관성) |

### 6-5. 보유 종목 패널
- 종목별 표시 정보:
  - 종목명, 종목코드
  - 보유 수량
  - 평균 매수가 / 현재가
  - 평가금액 / 평가손익
  - 수익률 (%)
- 현재가는 해당 턴 날짜의 종가 (또는 가장 가까운 이전 영업일 종가)

### 6-6. 잔고 패널
- 보유 현금
- 평가 금액 (보유 종목 총 시가)
- 총 자산 (현금 + 평가금액)
- 수익률 (시드머니 대비)

### 6-7. 그룹 멤버 수익률 랭킹
- 참여자별 실시간 계산 (현재 턴 날짜 기준 종가로 보유 종목 평가)
- 표시: 순위, 이름, 총자산, 수익률
- 총자산 기준 내림차순 정렬

### 6-8. 그룹 채팅
- 다른 팀원이 개발하는 그룹 채팅 모듈 임베드 예정
- 채팅 완성 전까지는 빈 영역에 플레이스홀더 표시
- 투표 UI도 채팅 연동 전까지는 모달로 표시 (6-1 투표 UI 구현 전략 참고)

### 6-9. 게임 종료
- 수동 종료: "게임 종료" 버튼 클릭 → 확인 팝업
- 자동 종료: 다음 턴 날짜가 게임 종료일 초과 시
- 종료 처리:
  1. 현재 활성 턴 COMPLETED 처리
  2. 모든 참여자 최종 자산/수익률 계산
  3. 마지막 턴 포트폴리오 스냅샷 저장
  4. 참여자 랭킹 확정 (총자산 기준)
  5. 게임 상태: **IN_PROGRESS → FINISHED**

## 7. 게임 결과

### 7-1. 최종 성과 요약
- 최종 자산 (원 단위)
- 최종 수익률 (%)
- 총 거래 횟수

### 7-2. 자산 변동 그래프
- X축: 게임 날짜, Y축: 총 자산
- 턴마다 저장된 포트폴리오 스냅샷 기반 라인 차트
- lightweight-charts LineSeries 사용

### 7-3. 나비 최종 분석 리포트 (AI)
#### 데이터 구성
- 게임 중 매 턴마다 저장된 AI 브리핑 (game_turn.ai_briefing)
- 날짜별 매매 내역 (종목명, 매수/매도, 수량, 가격)
- 날짜별로 시장 동향 + 해당 날 매매를 매칭하여 GPT에 전달

#### GPT 프롬프트 구조
```
[2024-01-05 시장 동향]
금리 인하 기대감에 코스피 상승...
→ 이 날 매매: 삼성전자 매수 10주 @70,000

[2024-01-12 시장 동향]
부동산 PF 우려 확산...
→ 이 날 매매 없음
```
→ "위 시장 상황과 매매 결정을 비교 분석하여 각 매매가 적절했는지 평가하세요"

#### 응답 형식
```
[투자 성과] (2문장)
[투자 패턴 분석] 시장 동향과 매매 결정을 연결하여 분석 (3~4문장)
[개선 제안] 시장 상황 대비 더 나은 결정이 가능했던 시점 언급 (2~3문장)
```
- 이모티콘 사용 금지
- AI 이름: "나비"
- **AI 모델**: gpt-4o (게임당 1회 호출, 매매 패턴 분석은 깊은 추론 필요)
- GPT 사전 지식 사용 금지 (제공된 브리핑 + 매매 데이터만 근거로 분석)

### 7-4. 전체 매매 내역 테이블
| 컬럼 | 설명 |
|------|------|
| 날짜 | 매매 발생 턴 날짜 |
| 종목 | 종목명 + 종목코드 |
| 구분 | 매수(빨강) / 매도(파랑) 뱃지 |
| 수량 | 체결 수량 |
| 단가 | 체결 가격 |
| 금액 | 수량 x 단가 |
