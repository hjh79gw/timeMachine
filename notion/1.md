# (필독) 개발 기능 명세서 작성 기준

## 1. 기본 원칙

- 기능명세서 1개 = Jira 이슈 1개
- 하나의 이슈는 완료/미완료를 명확히 판단할 수 있어야 한다
- 기능은 UI 단위가 아니라 사용자 목적 및 업무 흐름 단위로 정의한다
- 하나의 이슈는 한 사람이 책임지고 끝낼 수 있어야 한다

---

## 2. 이슈 크기 기준

다음 기준을 모두 만족하는 것이 이상적인 이슈 크기이다.

### 시간 기준

- 0.5일 ~ 3일 이내 완료 가능해야 한다
- 4일 이상 예상 시 반드시 분리한다

### 완료 정의 기준

- 기능이 끝났는지 명확히 판단 가능해야 한다

좋은 예:

- 주식 매수 API 구현
- 뉴스 조회 API + 페이징

나쁜 예:

- 뉴스 기능 구현
- 투자 시스템 만들기

---

### PR 기준

- 하나의 이슈는 PR 1~2개 수준이 적절하다
- PR이 3개 이상 예상되면 이슈를 분리한다

---

### 책임 범위 기준

- 한 명이 독립적으로 수행 가능해야 한다
- 협업 의존도가 높다면 기능을 분리한다

---

### 흐름 단위 기준

기능은 단계별 업무 흐름 단위로 분리한다.

예: 뉴스 서비스

- 뉴스 수집
- 뉴스 정제/태깅
- 임베딩 생성
- 벡터 DB 저장
- 뉴스 조회 API
- 뉴스 요약 생성

각 단계는 독립적으로 완료 가능해야 한다

---

## 3. 이슈 분리 조건

다음 중 하나라도 해당하면 이슈를 분리한다.

- 예상 작업 기간이 4일 이상인 경우
- PR이 3개 이상 발생할 것으로 예상되는 경우
- 완료 기준이 모호한 경우
- 기능이 여러 단계 흐름을 포함하는 경우
- 협업 의존도가 높은 경우
- 테스트 또는 배포를 독립적으로 해야 하는 경우

---

## 4. 작성 시 체크 기준

이슈 생성 전 아래 항목을 확인한다.

- 3일 이내 완료 가능한가
- 완료 여부를 명확히 판단할 수 있는가
- PR 2개 이하로 끝날 수 있는가
- 한 사람이 책임지고 수행 가능한가
- 다른 기능과 독립적으로 동작하는가

위 항목 중 하나라도 충족하지 못하면 이슈를 분리한다
